# [분류 카테고리]

## meeting_log (회의록) — 정상 처리 대상
- 2인 이상의 발화자가 식별 가능 (예: `**PM:**`, `**개발자:**`, 또는 "○○○이 말했다")
- 의사결정/토론/논의 내용 포함
- 일시/참석자 정보가 명시 또는 추론 가능
- 예: 킥오프 미팅, 스프린트 플래닝, 기술 의사결정 회의, 인터뷰 정리

## retrospective (회고록 / 포스트모템) — 경고 후 처리
- 단일 작성자 시점의 회고 형식
- "좋았던 점 / 아쉬웠던 점 / 다음 액션" 같은 구조
- Sprint Retro, Post-mortem, KPT, 5whys 등
- 회의록은 아니지만 software 맥락은 있음

## spec_doc (명세서 / 기획서) — 경고 후 처리
- API 명세 / 데이터 모델 / PRD-like 문서
- 이미 정리된 요구사항/기획안
- "Harness 의 CPS → PRD 단계를 거꾸로 가는 셈" — 권장 X 지만 진행은 가능

## task_request (작업 요청) — 거부
- "엑셀로 매출 보고서 만들어줘"
- "워드 문서 양식 작성해줘"
- "이 페이지를 PDF 로 변환해줘"
- 회의/논의 내용이 아닌 단순 작업 지시

## general_doc (일반 문서) — 거부
- 회사 정책 / 가이드라인 / 위키 / 공지사항 / FAQ
- 개인 메모 / 일기 / 일정표
- 책 요약 / 강의 노트 등
- 소프트웨어 개발 맥락 없는 일반 문서

## unknown (불명) — 거부
- 위 어디에도 명확히 속하지 않음
- 너무 짧거나 의미 추출 불가
- 다른 언어 / 임의의 텍스트

# [판단 가이드]

1. **2인 이상 발화자**가 명확히 보이면 → 거의 확실히 `meeting_log`
2. 단일 1인칭 "나는 ...", "우리는 ..." 형식 + 회고적 어투 → `retrospective`
3. 항목/리스트로 구조화된 사양 (예: "API: GET /users → ..." / "필드 정의:") → `spec_doc`
4. "...해줘", "...만들어줘" 같은 요청문 위주 → `task_request`
5. 위 어디에도 안 맞고 일반 정보 전달 문서 → `general_doc`
6. 정말 모르겠으면 → `unknown` (보수적으로 거부)

**중요**: 페이지가 회의록 같은데 발화자가 모호하면 → `meeting_log` 보단 `retrospective` 로 분류 (낮은 confidence 부여). 잘못된 `meeting_log` 분류로 LLM 환각이 일어나는 게 더 나쁨.

# [출력 — JSON 객체만]

```json
{
  "type": "meeting_log" | "retrospective" | "spec_doc" | "task_request" | "general_doc" | "unknown",
  "confidence": 0.0~1.0,
  "reason": "<한 줄 한국어 근거 — 어떤 신호로 그 분류를 내렸는지>"
}
```

- 마크다운 fence (` ``` `) 없이 순수 JSON 객체만 출력
- confidence 는 솔직하게. 애매하면 0.5~0.7. 명확하면 0.9+.
- reason 은 사용자가 보고 납득할 수 있게 구체적으로 (예: "발화자 4명 식별, 의사결정 흐름 명확")
